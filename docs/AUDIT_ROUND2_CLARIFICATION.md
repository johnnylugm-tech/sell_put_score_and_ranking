### Bug 1：無財報預設值（有效 claim，診斷有誤）

#### 審計聲稱
> 所有無財報資料的股票會被 `days_to_earnings = 3` 誤標為禁止新倉

#### 實際數據（2026-04-17）
```
MU    days=68  forb=False  earnings=有
AMD   days=18  forb=False  earnings=有
TSM   days=-2  forb=False  earnings=有
QQQ   days=999 forb=False  earnings=無（QQQ特殊處理）
...
```

**16 檔全部有真實財報日期，0 檔被誤標。**

審計的診斷是**錯誤陳述**——邏輯漏洞是真實的（「如果加一檔查不到財報的新股票會觸發」），但「所有無財報資料的股票被誤標」不是事實。

#### 已修復內容
- `core.py`：`else 3` → `else 20`（20天 → s7=3，又不觸發 `0<=20<=7` 的禁止條件）
- 結果：無財報新品種 → days=20 → s7=3（保守）→ 0<=20<=7=False（不禁止）✅

### Bug 2：TSM 已過財報得 s7=0（審計錯誤，代碼正確）

#### 審計聲稱
> Doc 寫「已過財報 → 5」，TSM s7=0 違反 Doc

#### Doc 原文
```
| ETF / 已過財報 | 5 |
```

#### 正確解讀
這一行描述的是**兩種不同情境**：
1. **ETF**：`days=999 → >30 → s7=5`（特殊處理）
2. **已過財報**：威脅已結束 → 無需懲罰 → s7=0

TSM 財報 4/16 已過 → 無未來事件風險 → s7=0

#### 代碼邏輯
```
days_to_earnings > 30 → s7=5
elif days_to_earnings >= 15 → s7=3
elif days_to_earnings >= 7 → s7=1
else → s7=0    # -2 落在這裡，正確
```

#### 結論
代碼行為完全正確，Doc 也從未說「已過財報要得 5 分」。審計對 Doc 的閱讀有誤。
